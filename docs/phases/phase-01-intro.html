Why Project Structure Matters

A well-organized project structure is like a well-organized workshop. When everything has its place, you spend less time searching and more time building. Professional Python projects follow conventions that make the code easier to understand, maintain, and share with others.

Virtual Environments Explained

A virtual environment is an isolated Python installation for your project. Think of it as a clean room where your project has its own set of tools, separate from other projects and your system Python.

Why Use Virtual Environments?

  • Isolation: Each project can have different versions of the same package
  • Reproducibility: Others can recreate your exact environment
  • Cleanliness: Your system Python stays uncluttered
  • Safety: Experiments don't affect other projects

Common Commands

# Create a virtual environment (you've already done this!)
python -m venv venv

# Activate it (Windows)
venv\Scripts\activate

# Activate it (macOS/Linux)
source venv/bin/activate

# When activated, your prompt shows (venv)
(venv) C:\Projects\Desktop Agent>

# Deactivate when done
deactivate
Tip: Always activate your virtual environment before working on the project. You'll know it's active when you see (venv) in your terminal prompt.

Dependencies with requirements.txt

The requirements.txt file lists all packages your project needs. This allows anyone to install the exact same dependencies with a single command.

Our Dependencies

Package Purpose Version
PyQt6 Desktop GUI framework - creates windows, buttons, layouts >=6.6.0
Pillow Image processing library - handles screenshots >=10.0.0

Installing Dependencies

# Install all dependencies from requirements.txt
pip install -r requirements.txt

# Install a single package
pip install PyQt6

# See what's installed
pip list

# Save current packages to requirements.txt
pip freeze > requirements.txt
Note: The >=6.6.0 syntax means "version 6.6.0 or higher". This gives flexibility while ensuring compatibility. For production apps, you might pin exact versions like ==6.6.1 for reproducibility.

Git Version Control

Git tracks changes to your code over time. It's like having unlimited "undo" with the ability to see exactly what changed, when, and why. Every professional software project uses version control.

Core Git Concepts

  • Repository (repo): Your project folder, tracked by Git
  • Commit: A snapshot of your code at a point in time
  • Staging area: Where you prepare changes before committing
  • Branch: A parallel version of your code (we'll use main)

Essential Git Commands

# Initialize a new repository
git init

# Check status - see what's changed
git status

# Stage files for commit
git add filename.py
git add .              # Stage all changes

# Commit with a message
git commit -m "Add initial project structure"

# View commit history
git log --oneline
Commit Message Tips: Write messages in imperative mood ("Add feature" not "Added feature"). Keep the first line under 50 characters. Be specific about what changed and why.

GitHub: Sharing & Collaboration

GitHub hosts your Git repository in the cloud. This provides backup, enables collaboration, and makes your project accessible from anywhere. It's where most open-source projects live.

Git vs GitHub

  • Git: The version control tool (runs locally)
  • GitHub: A cloud service that hosts Git repositories

GitHub Workflow

# 1. Create a repo on GitHub (through the website)

# 2. Connect your local repo to GitHub
git remote add origin https://github.com/username/desktop-agent.git

# 3. Push your code to GitHub
git push -u origin main

# 4. After making changes locally:
git add .
git commit -m "Your message"
git push

# Pull changes from GitHub (if you edited online)
git pull
The -u flag: Using -u origin main the first time sets up tracking. After that, you can just use git push and git pull without extra arguments.

The .gitignore File

Not everything should be tracked by Git. The .gitignore file tells Git which files and folders to ignore. This keeps your repository clean and avoids sharing sensitive or unnecessary files.

What to Ignore

  • Virtual environment: venv/ - others will create their own
  • Python cache: __pycache__/, *.pyc - generated files
  • IDE settings: .vscode/, .idea/ - personal preferences
  • Secrets: .env, API keys - never commit these!
  • Screenshots: screenshots/ - user-generated content
# Example .gitignore for Python projects
venv/
__pycache__/
*.pyc
*.pyo
.env
.vscode/
.idea/
screenshots/
*.log

The __init__.py File

The __init__.py file marks a directory as a Python package. This allows you to import modules from that directory. It can be empty or contain initialization code. 

# src/desktop_agent/__init__.py
# This file can be empty - its presence makes desktop_agent a package

# Or it can define what's exported:
__version__ = "0.1.0"
__author__ = "Your Name"

With this structure, you can run the application using: 

python -m desktop_agent.main
