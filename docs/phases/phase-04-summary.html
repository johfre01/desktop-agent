Mouse Events Used

Event When It Fires Our Use
mousePressEvent Mouse button pressed down Store start position, determine action (drag vs resize)
mouseMoveEvent Mouse moves (always, with tracking) Update cursor on hover, execute drag/resize while button held
mouseReleaseEvent Mouse button released Clear stored state, end operation

Problem Solved: Transparent Click-Through

The Problem

Fully transparent areas don't receive mouse events - clicks pass through to windows below. This meant dragging from the center didn't work initially.

The Solution

Fill the window with a nearly-invisible color (QColor(0, 0, 0, 1) - alpha of 1 out of 255). This is invisible to the human eye but gives the window "substance" to capture mouse events.

# In paintEvent - fill with nearly invisible color
painter.fillRect(self.rect(), QColor(0, 0, 0, 1))

Key Takeaways

1. Hit Testing Determines Action

By checking where a click occurs (edge, corner, or center), we can decide whether to resize or drag. This pattern is called "hit testing" and is fundamental to custom UI interactions.

2. Store State on Press, Act on Move

The pattern is: store starting position and geometry on mouse press, calculate delta on each move, apply changes. This ensures smooth, predictable dragging and resizing.

3. Mouse Tracking for Hover Effects

setMouseTracking(True) enables mouseMoveEvent even when no button is pressed. This is essential for updating cursors based on hover position.

4. Resizing from Left/Top is Different

When resizing from right/bottom, only size changes. But from left/top, both position AND size must change to keep the opposite edge stationary. This is a common gotcha!

5. Always Respect Minimum Size

Check against minimumWidth() and minimumHeight() before applying new geometry to prevent the window from becoming unusably small.

"The best UI feels like it's reading your mind - but it's really just good hit testing."
